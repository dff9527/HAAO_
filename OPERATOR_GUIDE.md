# HAAO 操作指南（給 PM / Sales 背景的你）

這份文件假設你懂技術的「大概」，但不寫程式、也不想碰指令。看完你會知道 HAAO 是什麼、你在裡面扮演什麼角色，以及每天打開它要做哪幾件事。
（工程細節在 `README.md` 和技術文章裡，這份刻意不碰那些。）

---

## 一分鐘搞懂 HAAO 是什麼

把 HAAO 想成一間**全部由 AI 組成的軟體開發小公司**，而你是這間公司的老闆兼產品負責人。

- 你用白話講出「我要做什麼功能」。
- 一個聰明但比較貴的雲端 AI（Claude）把你的需求**拆成一張張小工單**。
- 一群便宜又快的本地 AI（跑在你電腦上的模型）**照工單寫程式、自己測試**。
- 整個流程的調度、催進度、出錯重派，由系統**自動**處理，不用你盯。
- 你只在**頭尾**出手：開工前核准、完工後驗收。

你不是工程師，你是**發包方 + 驗收方**。這就是整個產品的核心。

---

## 你的角色：Product Owner（產品負責人）

用招聘的比喻最好懂：

| 角色 | 是誰 | 白話職責 | 像公司裡的誰 |
|---|---|---|---|
| **Product Owner** | **你（人）** | 決定要做什麼、排優先、開工核准、完工驗收 | 老闆 / 產品負責人 |
| **Tech Lead** | 雲端 Claude | 把你的需求拆成可執行的小工單、最後做技術把關 | 資深技術主管 |
| **Scrum Master** | 系統本身（自動） | 派工、催進度、出錯重派——全自動 | 專案經理（但是機器人） |
| **開發團隊** | 本地 AI 模型 | 真正寫程式、跑測試的工人 | 工程師們 |

**重點**：那些瑣碎的「派工、移動工單、出錯重來」是系統自動做的，不需要你。你要握住的是「**這值不值得做、做得好不好**」——這才是你的價值，也是不能交給 AI 的部分。

## 你每天實際在做的四件事

整個使用流程就是下面四步，對應畫面上的操作：

### ① 提需求（在「需求編輯器 / Prompt Composer」）
點右上角「+ New requirement」，用白話寫你要的東西。例如：
> 「強化密碼儲存安全，登入時要擋掉太弱的密碼。」

你也可以順手填：要動哪個專案範圍、有什麼不能碰的限制（紅線）、優先級高低。
寫完按 **Decompose（拆解）**。

### ② 核准拆解（這是你的第一個關卡 Gate 1）
系統會把 Claude 拆好的工單**先給你看預覽**——通常是 3 張左右的小工單，例如「改用 bcrypt 加密」「登入擋弱密碼」「補上測試」。

你在這裡：看一下拆得合不合理、不要的可以勾掉、想改的可以改，然後按 **Approve & create（核准建立）**。
**沒按核准前，AI 不會真的開始動工。** 這道關卡就是防止 AI 自己想做什麼就做什麼。

### ③ 看進度（在 Kanban 看板）
工單核准後會自動開始跑。你在看板上看它們從左往右流動：

```
Backlog → Ready → In Progress → Review → Awaiting acceptance → Done
（待辦）  （就緒） （進行中）    （審查）  （等你驗收）        （完成）
```

點任何一張工單可以展開細節，看到 AI **即時的工作日誌**（像看著工人邊做邊回報：讀檔 → 改程式 → 跑測試 → 失敗了 → 自己重試…）。這就是 HAAO 跟「黑箱 AI」最大的差別：**你看得到它在幹嘛**，不是丟進去等結果。

這一段你基本上**不用做事**，看著就好。出錯它會自己重試，重試幾次還不行才會升級求救（自動找雲端 Claude）。

### ④ 驗收（你的第二個關卡 Gate 2）
工單通過 AI 的技術把關後，**不會直接算完成**，而是停在「Awaiting acceptance（等你驗收）」。
你看一下成果，覺得 OK 就按 **Accept（接受）** → 進 Done。
覺得不對，按 **Reject（退回）** 並寫一句原因，它會帶著你的意見重做。

就這樣。**你的工作就是頭尾兩關 + 中間看戲。**

---

## 三個主要畫面導覽

- **Kanban 看板**：主畫面，看所有工單的狀態流動。每張卡上有「哪個 AI 在做」「重試幾次了」「測試過了沒」。
- **工單細節面板**：點卡片展開，看這張工單要改什麼、怎麼算完成、以及 AI 的即時日誌。你的 Accept / Reject 按鈕在這裡。
- **Models（模型設定）頁**：管理你有哪些 AI 可用、哪個 AI 負責哪種工作、各自的參數。平常不太需要動，設定好一次就好。

## 名詞白話對照表

碰到這些詞不用緊張，這裡是人話翻譯：

| 看到的詞 | 白話意思 |
|---|---|
| **Ticket（工單）** | 一個小到不能再小的開發任務，例如「改一個函式」 |
| **Backlog** | 待辦清單 |
| **Decompose（拆解）** | 把你一句大需求，切成好幾張小工單 |
| **DoD / Definition of Done** | 「怎樣才算做完」的標準——通常是「某個測試要通過」 |
| **Retry（重試）** | AI 做錯了，自己再試一次（有次數上限） |
| **Escalate（升級）** | 本地 AI 試到上限還搞不定，自動找更強的雲端 AI 來救 |
| **Audit（技術審查）** | 雲端 Claude 檢查程式碼有沒有真的達標（自動，不用你做） |
| **Acceptance（驗收）** | 你（人）最後拍板「這就是我要的」 |
| **Local / Cloud（本地 / 雲端）** | 本地 = 跑在你電腦上、免費又快；雲端 = 比較聰明但要花錢 |
| **Model（模型）** | 一個 AI。HAAO 同時用好幾個，分工合作 |

## 什麼時候該你出手，什麼時候放著

**放著讓它跑（不用管）**：工單在 In Progress 跑、測試失敗自己重試、自動升級雲端——這些都是設計好的自動流程，盯著反而浪費你時間。

**該你出手**：
- 開工前（Gate 1）：拆解預覽要你核准。
- 完工後（Gate 2）：成果要你驗收。
- 一張工單變成 **Blocked（卡住）**：代表 AI 試到上限、升級也沒解決，需要你看一下——通常是需求講得不夠清楚，回去把需求拆小一點、講白一點再來一次。

---

## 常見狀況

- **「拆出來的工單怪怪的」** → 在 Gate 1 直接勾掉不要的，或回去把需求寫得更具體（指定範圍、講清楚不要動什麼）。AI 拆得好不好，很大程度取決於你需求寫得清不清楚。
- **「一直卡在某張工單」** → 那張任務可能太大。HAAO 的甜蜜點是「小到一次只改一個地方」。把它當成一個訊號：需求要再切細。
- **「我想換哪個 AI 做某類工作」** → 去 Models 頁的角色分配表改，或在單張工單細節裡臨時指定（單張的設定優先）。
